# Design: Governed Codex Dispatch Seam

- **Status:** Draft
- **Area:** Runtime / AFK Loop
- **Related Issues:** #116, #90, #99, #114, #115

## Context
Currently, the Codex agent (`chatgpt-codex-connector`) in the repository is triggered via an `@codex` comment on issues. This path bypasses the strict Demerzel governance loop established for Jules in ADR 0004. Unlike Jules, which uses a gated `ready-for-agent` label and respects the `governance/state/afk-halt.json` kill switch, Codex is ungoverned, un-haltable, and exposes the ecosystem to uncontrolled execution by any public commenter.

The goal of this design is to bring Codex into the exact same governance model as Jules, ensuring strict maintainer gating, haltable execution, fallback routing, and rigorous output review.

## Architecture & Workflow

The new Codex dispatch seam will replace the ad-hoc comment trigger with a robust GitHub Actions workflow centered around label-based gating.

### 1. Maintainer Gating via Label
- The `@codex` comment trigger will be strictly ignored or disabled.
- Maintainers with triage/write access will apply a new label: `codex-ready`.
- Only authorized contributors can apply this label, acting as an explicit, auditable gate.

### 2. Governance Halt Verification
- The dispatch workflow will listen for the `pull_request` or `issues` event matching the `codex-ready` label.
- The workflow **must** read the `governance/state/afk-halt.json` file.
- If the file exists and is not expired, the workflow stops execution, leaves a comment explaining that agent dispatch is currently halted, and retains the label for manual retry later.

### 3. Risk Policy & Allowlist
- A centralized risk policy (defined in `examples/agents/codex-dispatch-policy.example.json`) dictates limits.
- The workflow checks the task complexity. High or medium-risk tasks enforce specific constraints (e.g., timebounds, scope).

### 4. Bounded Execution & Skill Injection
- The workflow constructs a prompt for Codex or creates a `codex-task` artifact.
- **Skill Injection (#114/#115):** Skills stored in `.claude/skills/` (like those for Matt Pocock's methodologies) are supplied to Codex either by directly mapping them into the task payload, concatenating them into the system prompt, or mounting them into the agent's workspace.

### 5. Fallback Behavior
- Codex is subject to strict quota limits.
- If Codex execution fails due to quota exhaustion, API rate limits, or unavailability, the workflow initiates a **Fallback Route**.
- The fallback will automatically remove the `codex-ready` label, apply the `ready-for-agent` label (to dispatch Jules), and append a comment: *"Codex quota unavailable. Falling back to Jules for task execution."*

### 6. Output Generation
- For low-risk tasks, Codex may push directly to a PR.
- For medium/high-risk tasks, Codex **must** open a **Draft PR** or leave a detailed execution plan as a comment for maintainer approval before generating code.

### 7. Review Gate
- Code generated by Codex is subject to the same `qa-verdict-dispatch.yml` Demerzel tribunal as Jules.
- Human code review and passing CI builds (`dotnet build && dotnet test`) remain mandatory before merging into `main`.

## Non-Goals
- Do not implement unreviewed Codex automation.
- Do not bypass Jules governance structures.
- Do not allow arbitrary issue commenters to trigger agents.
