# AGENTS.md — tars

F# neuro-symbolic agent framework. Part of the four-repo GuitarAlchemist ecosystem (`tars` + `ga` + `ix` + `Demerzel`). **Role in the ecosystem:** cross-model theory validator — TARS is used by `ga` to cross-check music-theory hypotheses generated by other models, and runs its own self-improvement loop (Workflow-of-Thought DSL + 7-step promotion pipeline). Full project layout in [`README.md`](./README.md). Agent-facing conventions in [`CLAUDE.md`](./CLAUDE.md).

## Build, test, verify

All active code lives under `v2/`.

```powershell
cd v2
dotnet build                                  # full F# build
dotnet test                                   # ~820 tests (4 Docker-gated)
dotnet format --verify-no-changes             # lint
```

Repo harness verification (Agent Blackbox `VERIFY_COMMAND`):

```powershell
pwsh Scripts/verify.ps1
```

The verify script builds + tests `v2/Tars.sln` (or `Tars.sln` fallback) and exits non-zero on failure. Do **not** claim success without a green `verify.ps1`.

## Agent Blackbox operating boundaries

Risk policy: [`agent-blackbox.policy.json`](./agent-blackbox.policy.json) (risk thresholds, blocked paths, one-way doors). Loop policy: [`agent-blackbox.loop-policy.json`](./agent-blackbox.loop-policy.json) (which paths the supervised loop may edit).

**Blocked paths** (never edited by an autonomous loop — they must go through human PR review): `.github/workflows/**`, `infra/production/**`, `billing/**`, `legal/**`, `security/**`.

**One-way doors** (changes require explicit sign-off because they're hard to reverse): `schemas/**`, `contracts/**`, `migrations/**`, `**/*.sln`, `**/*.fsproj`, `Directory.Build.props`.

The PR workflow at [`.github/workflows/agent-blackbox.yml`](./.github/workflows/agent-blackbox.yml) runs `analyze`, emits a risk report + harness audit + response-quality report, comments them on the PR, uploads artifacts, and calls `enforce --report` to fail PRs whose verdict is `block` unless the `agent-blackbox-reviewed` label has been applied by a human reviewer.

## Review independence

The loop is allowed to *propose* but not *self-certify*. Independence is enforced by four mechanisms — each one rated separately by `install-audit`:

- **Producer-reviewer separation.** The generator (loop, agent, suggestion) is never the reviewer (`enforce`, human, tribunal). Author and reviewer roles are distinct in every PR; `dist/risk-report.json` is generated automatically and reviewed by a human (or by a different agent) before merge.
- **Fresh evaluator.** The PR check runs in a fresh sub-agent / fresh session with a different context window from the one that authored the change. The author cannot self-certify; that's why `enforce` runs as a separate workflow step rather than inside the producer.
- **Cross-vendor review.** Risk decisions on contested PRs may be re-run through a cross-vendor / multi-model / multi-LLM panel (Gemini, Codex peer, Demerzel tribunal). This is the same pattern documented in the cross-repo `docs/contracts/qa-verdict.contract.md` family.
- **Rewrite budget.** Autonomous edits respect a line budget (max lines / lines-per-fix / diff budget) to keep blast radius small. Larger changes must be split into multiple PRs so each one stays under the rewrite budget.
