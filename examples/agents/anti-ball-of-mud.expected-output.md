# Architecture Entropy Report

Generated by `anti-ball-of-mud-guard` for Issue #832.

**1. Entropy finding**
The requested change attempts to inject `Octokit` (GitHub API SDK) directly into `CoreOrchestrator.fs`. This is muddy because it mixes core business logic (orchestration) with external IO concerns (network, provider SDKs).

**2. Design vocabulary**
- **Adapter**: We lack an adapter for the GitHub API. The core module should not know about `Octokit`.
- **Narrow Seam**: We need a narrow seam between the orchestrator and the issue fetcher.
- **Deep Module**: The orchestrator should remain a deep module, dealing with domain types rather than HTTP responses.

**3. Proposed seam**
Introduce an `IIssueProvider` interface (or a record of functions like `fetchIssue: IssueId -> Result<Issue, Error>`) in the core domain. The actual GitHub implementation should live in an infrastructure adapter module (e.g., `Infrastructure/GitHubIssueAdapter.fs`).

**4. Smallest safe change**
Create the F# domain type `Issue` and the interface `IIssueProvider` in `v2/src/Domain/IssueContext.fs`. Implement a dummy/in-memory version of this provider and inject it into the orchestrator.

**5. Files likely touched**
- `v2/src/Domain/IssueContext.fs` (new)
- `v2/src/Core/Orchestrator.fs` (update constructor/injection)

**6. Test surface**
The seam can be tested by injecting a mock `IIssueProvider` that returns predetermined `Result.Ok` or `Result.Error` values, allowing us to unit test the orchestrator's behavior without making real network calls.

**7. Risk/backpressure**
I am stopping here because implementing the actual GitHub API adapter and wiring it into the dependency injection container is an architectural decision that expands the scope beyond a simple feature addition. It risks leaking secrets or creating a tight coupling to GitHub if not designed carefully.

**8. Review gate**
Human or Demerzel approval is required to confirm this proposed seam (`IIssueProvider`) and the subsequent infrastructure adapter before proceeding with implementation.
